# V14 Reference

How the DCC system works on FoundryVTT V14. This is an FYI reference for ongoing maintenance — not a migration guide. V14 Stable 1 (Build 359) was released April 1, 2026; the DCC system now targets V14 only.

## Compatibility

- `system.json` sets `{"minimum":"14","verified":"14","maximum":"14"}`
- V14 requires **Node.js 24** (not compatible with V13's Node.js requirement)
- No separate V13 compatibility branch is maintained

## Data Models (TypeDataModel)

The system uses `TypeDataModel` classes instead of `template.json`. Type names are declared in `system.json` under `documentTypes`, and classes are registered on `CONFIG.Actor.dataModels` / `CONFIG.Item.dataModels` in the `init` hook.

### File Layout

```
module/data/
├── fields/            # Reusable field classes (AbilityField, CurrencyField, DiceField, SaveField)
├── actor/             # BaseActorData + PlayerData, NPCData, PartyData
└── item/              # BaseItemData, PhysicalItemData, and per-type data models
```

### Adding a New Field

1. Add to the relevant `defineSchema()` return.
2. If the field shape is reused, promote it to a class in `module/data/fields/`.
3. If the change is not backwards-compatible with stored worlds, implement `static migrateData(source)` on the data model and convert legacy shape → new shape.

### Declaring Types in `system.json`

Type names must appear in `documentTypes` *before* `init` runs, or Foundry rejects documents with `"<type>" is not a valid type`. Keys here must match the keys used when registering data models.

```json
{
  "documentTypes": {
    "Actor": { "Player": {}, "NPC": {}, "Party": {} },
    "Item":  { "weapon": {}, "armor": {}, "spell": {}, ... }
  }
}
```

## Active Effects V2

### Required init setup

```javascript
CONFIG.ActiveEffect.legacyTransferral = false
CONFIG.ActiveEffect.phases = {
  initial: { priority: 0, label: 'Initial' },
  final: { priority: 100, label: 'Final' }
}
CONFIG.ActiveEffect.documentClass = DCCActiveEffect
```

Skipping `phases` causes `"initial" is not a registered ActiveEffect application phase`.

### Custom `DCCActiveEffect` document class

`module/active-effect.js` overrides `apply()` to handle three DCC-specific concerns:

1. **Equipped-item filtering** — effects on unequipped items are ignored.
2. **Dice chain effect type** — custom `type: 'diceChain'` moves a die along the DCC chain (d3→d4→d5→d6→d7→d8→d10→d12→d14→d16→d20→d24→d30). Value is signed step count.
3. **Signed-string numeric operations** — thief skills store numbers as `"+5"` / `"-2"` strings. V14's `DataField.applyChange()` treats these as strings (`"0" + "1" = "01"`), so the override intercepts add/multiply/upgrade/downgrade/override before calling `super.apply()` and re-formats with a sign prefix.

Do **not** override `applyActiveEffects` on the actor — V14's two-phase system will double-apply effects.

### Change-type values (strings, not numbers)

`CONST.ACTIVE_EFFECT_MODES` is deprecated. Use string `type` values on `change` objects:

| Old mode | New type |
|----------|----------|
| CUSTOM (0) | `'custom'` |
| MULTIPLY (1) | `'multiply'` |
| ADD (2) | `'add'` |
| DOWNGRADE (3) | `'downgrade'` |
| UPGRADE (4) | `'upgrade'` |
| OVERRIDE (5) | `'override'` |

The `applyActiveEffects` method accepts both with a fallback: `const type = change.type || change.mode || 'add'`.

### Effect targets

Target modifier fields so effects stack without clobbering base values:

| Attribute | Use |
|-----------|-----|
| Saves | `system.saves.{frt,ref,wil}.otherBonus` |
| AC | `system.attributes.ac.otherMod` |
| Initiative | `system.attributes.init.otherMod` |

### Effects compendium format

Stored in `packs/dcc-effects/`:
- `_id` must be exactly 16 alphanumeric characters
- `_key` follows `!effects!{_id}`
- `type` uses lowercase strings (`"add"`, not `2`)

## API Changes Worth Remembering

These V14 deprecations have been applied across the codebase — flag regressions if you see the old patterns:

| Deprecated | Current |
|------------|---------|
| `CONST.ACTIVE_EFFECT_MODES` | `CONST.ACTIVE_EFFECT_CHANGE_TYPES` (string values on `change.type`) |
| `foundry.utils.duplicate()` | `foundry.utils.deepClone()` |
| `foundry.utils.objectsEqual` | `foundry.utils.equals()` |
| `game.settings.get('core', 'rollMode')` | `game.settings.get('core', 'messageMode')` |
| `ChatMessage.applyRollMode()` | `ChatMessage.applyMode()` |
| `Roll.toMessage({ rollMode })` | `Roll.toMessage({ messageMode })` |
| `mergeObject({ performDeletions: true })` | `mergeObject({ applyOperators: true })` |
| `mergeObject({ '-=key': null })` | `foundry.data.operators.ForcedDeletion` |
| `CONFIG.statusEffects[].icon` | `CONFIG.statusEffects[].img` |
| `foundry.prosemirror.defaultPlugins` | `ProseMirrorEditor.buildDefaultPlugins()` |
| `TextEditor.implementation.enrichHTML()` | `TextEditor.enrichHTML()` (V14 re-exported at top level) |
| `Math.clamped()` | `Math.clamp()` |

### Roll mode string values

| V13 | V14 |
|-----|-----|
| `'publicroll'` | `'public'` |
| `'gmroll'` | `'gm'` |
| `'blindroll'` | `'blind'` |
| `'selfroll'` | `'self'` |

### Boolean values in Rolls

Booleans in dice formulas now evaluate to `0` / `1` automatically — no explicit coercion needed.

### `game.activeTool` and `SceneControls#tool`

Both can be `null` in V14 if no tools exist in the current layer. Null-check before use. DCC registers two custom scene control buttons (Fleeting Luck, Spell Duel) with `button: true`; these have not needed the new `interaction` / `control` / `creation` properties, but this is worth re-verifying if we add more tools.

### `updateSource` operator keys

The special keys `-=key` (delete) and `==key` (force overwrite) on `updateSource` are removed in V14. Use `DataFieldOperator` values instead. No production usages existed in DCC when we migrated.

## New V14 Features Available

- **Scene Levels** — vertically stacked images at defined elevations. Not used by DCC yet; a candidate for multi-level dungeons.
- **Pop-out ApplicationV2 windows** — all DCC sheets are ApplicationV2 and support this automatically.
- **Active Effect duration units** — beyond seconds: `years`, `months`, `days`, `hours`, `minutes`, `rounds`, `turns`. Combat expiry events: `combatStart`, `roundStart`, `turnStart`. DCC currently reads `effect.duration.seconds` only.
- **Batch document updates** — `Document.updateDocuments([...], { transaction: true })` for atomic multi-doc operations.
- **Font Awesome 7.2.0** — available for icon classes.

## Known Issue

Roll visibility: player-visible rolls can show as `'???'` when message mode is `gm` or `blind`; GM sees correct results. Text messages work normally. Reported to Foundry; no workaround on our side.
